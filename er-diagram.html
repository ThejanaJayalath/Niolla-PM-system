⇄ Entity Relationships
CLIENT ──< PROJECT 1 : N has many projects
PROJECT ──< PAYMENT_PLAN 1 : N has flexible plans
PAYMENT_PLAN ──< INSTALLMENT 1 : N scheduled (3–6)
INSTALLMENT ──< PAYMENT_TRANSACTION 1 : N partial or full payments
PAYMENT_TRANSACTION ──║ INVOICE 1 : 1 auto-generates receipt
CLIENT ──< NOTIFICATION 1 : N SMS / Email alerts
INSTALLMENT ──< NOTIFICATION 1 : N due / overdue triggers
ROLE ──< USER 1 : N admin / accountant / support
USER ──< AUDIT_LOG 1 : N tracks every action
USER ──< PAYMENT_TRANSACTION 1 : N recorded_by staff
PAYMENT_GATEWAY ──< PAYMENT_TRANSACTION 1 : N online payments (optional)
USER ──< REPORT 1 : N generates monthly reports
💡 Auto-Calculations remaining_balance in PAYMENT_PLAN auto-updates via trigger after each PAYMENT_TRANSACTION. overdue_days in INSTALLMENT computed daily via scheduled job.
⚠️ Partial Payment Handling A single INSTALLMENT can have multiple PAYMENT_TRANSACTIONs. Status flips from partial → paid when cumulative paid_amount ≥ due_amount.
✅ Role-Based Access ROLE.permissions stores JSON array of allowed routes/actions. Admin has full CRUD; Accountant limited to payment + invoice modules; Support is read-only with notification send access.
📤 Export & Integration REPORT.export_path stores generated XLSX/PDF file references. PAYMENT_GATEWAY.gateway_id is nullable in PAYMENT_TRANSACTION — cash/bank payments don't need a gateway record.
